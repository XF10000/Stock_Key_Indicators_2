## 1. 阶段完成标记流程

### 1.1 完成标记步骤

```
阶段开发完成
    │
    ├── 1. 代码自检
    │     ├── 运行 black 格式化
    │     ├── 运行 flake8 检查
    │     ├── 运行 mypy 类型检查
    │     └── 运行单元测试
    │
    ├── 2. 提交代码
    │     ├── git add .
    │     ├── git commit -m "feat: 完成阶段X - [阶段名称]"
    │     └── git push origin main
    │
    ├── 3. 创建Git标签
    │     └── git tag -a vX-[阶段名称]-YYYYMMDD -m "阶段X完成"
    │
    ├── 4. 更新文档
    │     ├── 更新 DEV_TRACKER.md 阶段完成记录
    │     ├── 更新 API.md 接口文档（如有变更）
    │     └── 更新 DEV_TRACKER.md 接口变更记录（如有变更）
    │
    ├── 5. 记录已知问题
    │     └── 在 DEV_TRACKER.md 技术债务记录中添加
    │
    └── 6. 通知相关人员
          └── 提交消息中 @相关人员
```

## 2. 新工程师接手流程

## 3. 接口变更管理

### 3.1 接口变更流程

```
发现接口变更需求
    │
    ├── 1. 评估影响范围
    │     ├── 哪些模块受影响？
    │     ├── 哪些测试需要更新？
    │     └── 是否需要同步更新文档？
    │
    ├── 2. 提出变更申请
    │     ├── 在 DEV_TRACKER.md 中记录变更计划
    │     └── 通知相关开发人员
    │
    ├── 3. 实施变更
    │     ├── 修改接口定义
    │     ├── 更新调用代码
    │     └── 更新测试代码
    │
    ├── 4. 更新文档
    │     ├── 更新 API.md
    │     ├── 更新架构设计文档（如需要）
    │     └── 在 DEV_TRACKER.md 中记录变更
    │
    └── 5. 验证变更
          ├── 运行所有测试
          ├── 确认无回归问题
          └── 提交代码
```

### 3.2 接口变更记录格式

| 字段 | 说明 | 示例 |
|------|------|------|
| 变更ID | 唯一标识 | CHG-001 |
| 变更日期 | 变更时间 | 2025-03-06 |
| 模块 | 变更模块 | ttm.py |
| 接口名称 | 变更接口 | TTCalculator.calculate() |
| 变更类型 | 变更分类 | 新增 / 修改 / 删除 / 废弃 |
| 变更原因 | 变更理由 | 支持多字段同时计算 |
| 变更内容 | 具体变更 | 新增 columns 参数，支持列表输入 |
| 影响范围 | 受影响模块 | engine.py, test_ttm.py |
| 向后兼容 | 是否兼容 | 是 / 否（需迁移） |
| 变更人 | 变更作者 | 张三 |
| 审核人 | 审核人员 | 李四 |

**记录示例**：

```markdown
| 变更ID | 变更日期 | 模块 | 接口名称 | 变更类型 | 变更原因 | 变更内容 | 影响范围 | 向后兼容 | 变更人 | 审核人 |
|--------|---------|------|---------|---------|---------|---------|---------|---------|--------|--------|
| CHG-001 | 2025-03-06 | ttm.py | TTCalculator.calculate() | 修改 | 支持多字段计算 | 新增 columns 参数 | engine.py | 是 | 张三 | 李四 |
```

### 3.3 接口变更通知机制

| 通知方式 | 适用场景 | 通知内容 |
|---------|---------|---------|
| Git提交消息 | 代码提交 | 变更内容、影响范围 |
| DEV_TRACKER.md | 记录存档 | 完整变更记录 |
| API.md | 文档更新 | 新接口定义 |
| 代码注释 | 废弃接口 | 废弃说明、替代方案 |

**Git提交消息格式**：

```
feat(ttm): 新增 columns 参数支持多字段计算

变更内容:
- TTCalculator.calculate() 新增 columns 参数
- 支持同时计算多个字段的TTM

影响范围:
- engine.py 需更新调用方式（向后兼容）

文档更新:
- API.md 已更新接口定义
- DEV_TRACKER.md 已记录变更

测试:
- test_ttm.py 新增多字段测试用例
```

**废弃接口标注格式**：

```python
def calculate_old(self, df: pd.DataFrame) -> pd.DataFrame:
    """计算TTM（已废弃）
    
    .. deprecated:: 1.1.0
        使用 calculate() 方法替代，支持多字段计算。
    
    Args:
        df: 原始数据DataFrame
    
    Returns:
        添加TTM列后的DataFrame
    """
    import warnings
    warnings.warn(
        "calculate_old() 已废弃，请使用 calculate() 方法",
        DeprecationWarning,
        stacklevel=2
    )
    return self.calculate(df)
```

## 5. 紧急交接流程

### 5.1 紧急交接场景

| 场景 | 说明 | 处理优先级 |
|------|------|-----------|
| 开发人员突发离职 | 无法完成正常交接 | P0 - 立即处理 |
| 开发人员生病/请假 | 临时交接，后续可能恢复 | P1 - 当日处理 |
| 项目紧急调整 | 需要快速切换任务 | P1 - 当日处理 |
| 关键Bug修复 | 需要他人临时接手 | P2 - 尽快处理 |

### 5.2 紧急交接步骤

```
紧急交接触发
    │
    ├── 1. 确认交接范围
    │     ├── 当前正在开发的模块
    │     ├── 未提交的代码
    │     └── 相关文档
    │
    ├── 2. 快速状态记录（15分钟内）
    │     ├── 当前任务进度
    │     ├── 未完成工作
    │     ├── 已知问题
    │     └── 关键代码位置
    │
    ├── 3. 代码状态处理
    │     ├── 提交已完成代码
    │     ├── 暂存未完成代码（git stash）
    │     └── 记录代码状态
    │
    ├── 4. 文档更新
    │     ├── 更新 DEV_TRACKER.md 状态
    │     ├── 记录紧急交接原因
    │     └── 标注待处理事项
    │
    └── 5. 接手人快速上手
          ├── 阅读紧急交接记录
          ├── 检查代码状态
          └── 继续开发或修复
```
