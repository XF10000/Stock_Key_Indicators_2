## 1. 技术方案

### 1.1 核心技术栈

| 类别 | 技术选型 | 版本 | 用途 |
|------|---------|------|------|
| 编程语言 | Python | 3.10+ | 主要开发语言 |
| ORM框架 | SQLAlchemy | 2.0+ | 数据库访问层 |
| 数据处理 | pandas | 2.0+ | 数据清洗、计算、转换 |
| 数值计算 | numpy | 1.24+ | 数值运算 |
| 数据库 | SQLite | 3.x | 数据存储（已存在） |
| 配置管理 | PyYAML | 6.0+ | 重构规则配置文件 |
| Excel输出 | openpyxl | 3.1+ | Excel文件生成 |
| 进度显示 | tqdm | 4.65+ | 批量处理进度条 |
| 命令行 | argparse | 内置 | CLI参数解析 |
| 日志 | logging | 内置 | 运行日志记录 |

### 1.2 技术选型理由

**Python 3.10+**
- 项目已有代码库基于Python 3.10
- 类型提示支持完善，配合mypy静态检查
- match-case语法便于处理复杂条件分支

**SQLAlchemy 2.0+**
- 项目已有ORM模型定义（`models.py`）
- 支持类型安全的查询构建
- 复合索引已建立（stock_code + report_date），查询性能有保障

**pandas 2.0+**
- 财务数据天然适合DataFrame结构
- 向量化计算性能优于循环
- 与SQLAlchemy无缝集成
- 支持Excel读写

**SQLite**
- 已有358MB数据库文件，无需迁移
- 单文件部署，无运维成本
- 读多写少场景，SQLite性能足够
- 支持索引优化

**openpyxl**
- 纯Python实现，无系统依赖
- 支持多Sheet、样式设置、公式
- pandas原生支持openpyxl引擎

**PyYAML**
- 重构规则需要配置化
- YAML格式可读性好，支持注释
- 便于非开发人员理解和修改

### 1.3 替代方案

| 方案 | 优缺点 | 结论 |
|------|--------|------|
| **数据库迁移** | PostgreSQL性能更好，支持并发 | 不推荐：迁移成本高，当前规模不需要 |
| **xarray替代pandas** | 更适合多维数据 | 不推荐：学习成本高，pandas已足够 |
| **polars替代pandas** | 性能更好 | 备选：若pandas性能不足可考虑 |
| **xlsxwriter替代openpyxl** | 写入性能更好 | 备选：若openpyxl性能不足可考虑 |

## 2. 数据规模估算

### 2.1 数据量估算

| 数据类型 | 数量估算 | 说明 |
|---------|---------|------|
| 股票数量 | 5,000+ | A股上市公司 |
| 报告期数量 | 40期 | 10年 × 4季度 |
| 资产负债表记录 | 200,000条 | 5,000 × 40 |
| 利润表记录 | 200,000条 | 5,000 × 40 |
| 现金流量表记录 | 200,000条 | 5,000 × 40 |
| **总记录数** | **600,000条** | 三表合计 |

### 2.2 存储空间估算

| 存储项 | 当前大小 | 预估增长 | 说明 |
|--------|---------|---------|------|
| 主数据库 | 358MB | +50MB/年 | 季度数据增量更新 |
| 输出Excel | 500KB/公司 | 2.5GB | 5,000公司 × 500KB |
| 日志文件 | 10MB/月 | 120MB/年 | 按月轮转 |
| 配置文件 | <1MB | - | 规则配置 |
| **总计** | **~3GB** | - | 含输出文件 |

### 2.3 性能瓶颈预估

| 瓶颈点 | 预估影响 | 风险等级 |
|--------|---------|---------|
| **单公司查询** | 3表 × 10年 = 30次查询，约100ms | 低 |
| **TTM计算** | 需要跨年度数据关联，约50ms | 低 |
| **重构计算** | 100+字段计算，约200ms | 低 |
| **Excel输出** | 3个Sheet × 100行 × 10列，约300ms | 中 |
| **批量处理** | 5,000公司 × 650ms ≈ 54分钟 | 中 |
| **内存占用** | 单公司数据 <10MB，批量处理需流式处理 | 低 |

**性能结论**：
- 单公司处理时间 <1秒，满足交互需求
- 批量处理需实现进度显示和断点续传
- 内存占用可控，无需特殊优化

### 3.2 潜在风险

| 风险 | 影响程度 | 发生概率 | 说明 |
|------|---------|---------|------|
| **数据缺失** | 中 | 高 | 部分公司部分字段为空，需处理空值 |
| **字段映射错误** | 高 | 低 | 已有完整映射配置，需验证 |
| **计算精度问题** | 低 | 中 | 浮点数精度，需统一处理 |
| **Excel格式兼容** | 低 | 低 | openpyxl兼容性好 |
| **批量处理中断** | 中 | 中 | 需实现断点续传 |
| **规则变更** | 中 | 中 | 配置化设计降低影响 |

### 3.3 应对措施

| 风险 | 应对措施 |
|------|---------|
| 数据缺失 | 空值输出 "-"，不中断处理；日志记录缺失情况 |
| 字段映射错误 | 单元测试验证映射正确性；与原始数据对比校验 |
| 计算精度问题 | 统一使用float64；百分比保留2位小数 |
| Excel格式兼容 | 使用openpyxl标准API；避免复杂公式 |
| 批量处理中断 | 实现进度文件；支持指定起始公司 |
| 规则变更 | YAML配置文件；版本化管理 |

## 5. 推荐方案

### 5.1 推荐技术栈

```
┌─────────────────────────────────────────────────────────┐
│                    命令行接口层                          │
│                    argparse + tqdm                      │
├─────────────────────────────────────────────────────────┤
│                    业务逻辑层                            │
│    重构规则引擎 + TTM计算 + 数据校验                      │
│                    (纯Python实现)                       │
├─────────────────────────────────────────────────────────┤
│                    数据访问层                            │
│              SQLAlchemy ORM + pandas                    │
├─────────────────────────────────────────────────────────┤
│                    数据存储层                            │
│          SQLite数据库 + YAML配置文件                     │
└─────────────────────────────────────────────────────────┘
```

### 5.2 推荐理由

1. **延续现有架构**：项目已有SQLAlchemy模型、pandas处理逻辑，无需重构
2. **技术成熟稳定**：所选库均为Python生态主流选择，文档完善
3. **性能满足需求**：单公司处理 <1秒，批量处理可接受
4. **易于维护**：配置化设计，规则变更无需改代码
5. **测试友好**：纯函数设计，易于单元测试

### 6.2 进度恢复机制说明

**场景分析**：批量处理5000家公司时，程序可能因异常中断。

**解决方案**：简单的进度日志 + 命令行参数

```bash
# 首次运行，记录进度
python -m analysis.restructure --codes all --years 10

# 中断后继续（从指定股票开始）
python -m analysis.restructure --codes all --years 10 --start-from 000333
```

进度日志格式：
```
[2025-03-04 21:00:00] 处理完成: 000001
[2025-03-04 21:00:01] 处理完成: 000002
...
[2025-03-04 21:05:00] 处理中断于: 000333
```
